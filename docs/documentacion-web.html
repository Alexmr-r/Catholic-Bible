Anatomía de UNA petición autenticada (el patrón que repiten TODOS los flujos)

Antes de ver los flujos uno a uno, entiende el esqueleto común. Toda petición con JWT pasa por estas piezas en este orden exacto:

📱 Pantalla (ej. FavoritesScreen)→ servicio TS (favorites.service.ts)→ api.client.ts añade "Bearer token"→ JwtAuthenticationFilter→ Controlador REST→ UseCase (interfaz)→ Service→ RepositoryPort (interfaz)→ PersistenceAdapter→ JpaRepository→ 🗄️ PostgreSQL

Y el detalle clave que el tribunal puede preguntar: ¿cómo sabe el backend QUIÉN eres?

  1. api.client.ts (app) lee el accessToken de AsyncStorage y lo mete en la cabecera Authorization: Bearer eyJhbG….
  2. JwtAuthenticationFilter.doFilterInternal() intercepta la petición ANTES de llegar al controlador: extrae el token de la cabecera, lo valida con JwtTokenProvider.validateToken() (firma + expiración + blacklist) y saca el email del claim.
  3. Con ese email llama a CustomUserDetailsService.loadUserByUsername(email) → que usa UserRepositoryPort.findByEmail() → carga el usuario de BD.
  4. Crea un UsernamePasswordAuthenticationToken y lo guarda en el SecurityContextHolder: a partir de aquí "hay sesión" durante esa petición.
  5. El controlador recibe el usuario con @AuthenticationPrincipal UserDetails userDetails y resuelve su UUID con el helper getUserId(userDetails) → userRepository.findByEmail(email).getId().
  6. Ese UUID userId se pasa al caso de uso: cada query filtra por user_id, así un usuario jamás ve datos de otro.
Para la defensa: si te preguntan "¿y si yo te mando un favorito con el id de otro usuario?" → imposible: el userId NUNCA viene del JSON del cliente, siempre se deriva del token firmado. Modificar el token rompe la firma HMAC y el filtro lo rechaza con 401.

6 · Seguridad: JWT y Spring Security

  • CSRF off, sesiones STATELESS (el servidor no guarda sesión: todo va en el token), BCrypt para contraseñas.
  • JwtAuthenticationFilter corre antes que el filtro estándar de Spring: Bearer → validar → email → cargar usuario → SecurityContext (detallado en la anatomía de una petición).
  • AccessToken 24 h (claims: sub=userId, email, name, type=access) · RefreshToken 7 días (sub, type=refresh). Firma HMAC-SHA con JWT_SECRET (Base64).
  • Logout: blacklist de tokens en memoria (Set) — se pierde al reiniciar (mejora: Redis/BD).
  • Rutas públicas en SecurityConfig: /auth/**, swagger, /actuator/health, POST /chat, /admin/**, GETs de biblia y lectura diaria. Todo lo demás autenticado.

7.1 Conceptos básicos, explicados con ejemplos DE ESTE proyecto

¿Qué es una tabla, una fila y una columna?

Una tabla es como una hoja de Excel: las columnas definen qué datos guardas (email, nombre…) y cada fila es un registro concreto (un usuario, un versículo). La tabla users tiene una fila por cada persona registrada; la tabla verses tiene una fila por cada versículo de la Biblia (¡decenas de miles!).

PRIMARY KEY (PK) — el DNI de cada fila

La clave primaria identifica de forma única cada fila: no puede repetirse ni ser nula. Casi todas las tablas usan id UUID PRIMARY KEY DEFAULT gen_random_uuid(): Postgres genera el UUID solo al insertar. Excepción interesante: books.id es un VARCHAR(50) legible ('genesis', 'matthew') — una clave natural, elegida porque la app construye URLs como /bible/books/genesis.

¿Por qué UUID y no 1, 2, 3 (autoincremento)? (1) No son adivinables: con ids 1,2,3 cualquiera podría iterar /users/1, /users/2… (2) Se pueden generar en el código Java (UUID.randomUUID()) sin preguntar a la BD — útil para el modo offline con IDs temporales. (3) No chocan si algún día hay varias bases de datos.

FOREIGN KEY (FK) — el pegamento entre tablas

Una clave foránea es una columna que apunta a la PK de otra tabla. Es lo que "conecta" las tablas. Ejemplo real:

CREATE TABLE verses (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    section_id UUID NOT NULL REFERENCES sections(id) ON DELETE CASCADE,
    ...
);

verses.section_id REFERENCES sections(id) significa: "cada versículo pertenece a una sección que debe existir". Postgres lo vigila (integridad referencial): si intentas insertar un versículo con un section_id inventado, da error; si intentas borrar una sección que aún tiene versículos, también (salvo que definas qué hacer, ↓).

UNIQUE — prohibido repetirse

Una restricción UNIQUE impide dos filas con el mismo valor. Puede ser de una columna o compuesta (la combinación no puede repetirse):

  • users.email UNIQUE → no puede haber dos cuentas con el mismo email (por eso el registro devuelve 409).
  • chapters UNIQUE(book_id, chapter_number, version) → "Génesis capítulo 1 en versión Jerusalem Bible" solo puede existir una vez, pero permite un futuro "Génesis 1 versión Vulgata".
  • highlights UNIQUE(user_id, book_id, chapter_number, verse_number) → un usuario solo tiene UN color por versículo (re-subrayar = reemplazar).
  • reading_history UNIQUE(user_id, daily_reading_id) → no puedes marcar la misma lectura como leída dos veces.
  • reading_progress UNIQUE(user_id, date) → un solo check de constancia por día.
  • daily_readings.date UNIQUE → una sola lectura litúrgica por fecha.
  • Curiosidad con historia: favorites TENÍA un UNIQUE igual que highlights y la migración V4 lo eliminó — decisión de producto para permitir guardar el mismo versículo más de una vez (p. ej. con notas distintas).

NOT NULL y DEFAULT

NOT NULL = la columna es obligatoria (no puede quedar vacía). DEFAULT = valor automático si no envías nada: active BOOLEAN NOT NULL DEFAULT true, created_at TIMESTAMP NOT NULL DEFAULT NOW(), is_premium … DEFAULT false (nadie nace premium).

INDEX — el índice del libro

Un índice es una estructura extra que acelera las búsquedas, como el índice alfabético de un libro: sin él, Postgres lee la tabla entera (escaneo secuencial). A cambio, ocupa espacio y ralentiza un poco los INSERT. Índices reales del proyecto:

  • idx_users_email → el login busca por email en cada petición: tiene que ser instantáneo.
  • idx_favorites_user, idx_writings_user, idx_highlights_chapter (user, book, chapter) → todas las pantallas filtran "lo MÍO": el índice por user_id evita recorrer los datos de todos los usuarios.
  • idx_reading_progress_user_date y …_user_month → el calendario pide "mis días de este mes".
  • idx_verses_text USING gin(to_tsvector('spanish', text)) → índice GIN full-text pensado para búsqueda lingüística. Honestidad técnica: la búsqueda actual usa LIKE y no lo aprovecha (y quedó configurado en 'spanish' cuando V6 pasó el contenido a inglés) — está en deuda técnica.

Cómo se traduce todo esto a Java (JPA)

tabla users↔ UserEntity.java (@Entity, @Table, @Column)↔ UserPersistenceAdapter (convierte)↔ User.java (modelo puro, sin anotaciones)

Cada tabla tiene su espejo *Entity en infraestructura con anotaciones JPA; el adapter convierte Entity↔Modelo de dominio. Así el dominio no sabe ni que existe SQL. Y el esquema NO lo crea Hibernate (ddl-auto: validate): lo crean las migraciones Flyway V1–V11, scripts SQL versionados e inmutables que se ejecutan en orden al arrancar.

16 · Deuda técnica y riesgos conocidos

1. /admin/** sin auth de servidor + login del backoffice hardcodeado en el cliente. Propuesta: rol ADMIN con JWT o API key.
2. POST /chat público → riesgo de abuso/coste del LLM. Propuesta: JWT + rate limiting.
3. Estado en memoria: blacklist de tokens y códigos de reset (sin TTL) se pierden al reiniciar. Propuesta: Redis o tabla con expiración.
4. Apple Sign-In sin verificación de firma.
5. Búsqueda LIKE que no usa el índice GIN (y el índice es 'spanish' con contenido en inglés tras V6).
6. SSL Flexible (tramo Cloudflare→Droplet en claro) · 7. CORS * por defecto · 8. JWT_SECRET con default en el yml · 9. dashboard-stats/audit-logs hacen findAll() en memoria (no escala) · 10. activeNow/aiQueries del dashboard son estimadores, no contadores.

17 · Preguntas y respuestas para la defensa

¿Por qué arquitectura hexagonal?

Aísla el dominio de los detalles técnicos. BibleService depende de la interfaz BibleRepositoryPort, no de JPA: cambiar PostgreSQL por MongoDB solo toca los adaptadores de salida. El dominio se testea sin BD.

¿Por qué no hay DTOs/mappers de aplicación?

Evitar el triple mapeo Domain↔AppDTO↔RestDTO sin valor en este tamaño de proyecto. El mapeo vive en la periferia: records en controladores y builders en persistence adapters. Menos clases, menos GC, más legible.

¿Qué es RAG y por qué lo usaste?

Retrieval-Augmented Generation: recupero 15 versículos reales de PostgreSQL y los inyecto en el prompt antes de generar; la IA cita escritura real y alucina menos. Paso 0 propio: un LLM extrae la keyword, corrige typos y la traduce al inglés porque el texto de la BD está en inglés (CPDV).

¿Cómo sabe el backend quién soy en cada petición?

El JwtAuthenticationFilter valida el Bearer, saca el email del claim, carga el usuario vía CustomUserDetailsService y lo pone en el SecurityContext; el controlador resuelve el UUID y toda query filtra por user_id. El userId nunca viene del JSON del cliente.

¿Cómo funciona el premium y el trial?

hasAccess = premium ∨ trial(7 días). Premium: entitlement de RevenueCat (SDK) o flag en BD desde el backoffice. Trial persistido por email en user_trials: borrar la cuenta no lo reinicia.

¿Por qué UUID en vez de ids autoincrementales?

No son adivinables/enumerables, se generan en el código sin ir a la BD (clave para el modo offline con IDs temporales) y no chocan entre entornos.

¿Cómo funciona el offline?

Biblia completa en un JSON descargado a FileSystem; caché de datos personales en AsyncStorage; cola de operaciones pendientes con IDs temporales que sync.service reconcilia al volver la conexión.

¿Cómo gestionas el esquema de BD?

Flyway con 11 migraciones inmutables y versionadas; Hibernate en validate. Cualquier entorno converge al mismo esquema de forma reproducible.

¿Por qué Expo y no React Native puro?

Velocidad: dev client + QR para probar en físico, EAS Build para compilar iOS/Android sin mantener toolchains nativos, y módulos (speech, filesystem, apple-auth) de primera mano.

¿Dónde está desplegado?

Backend en un droplet de DigitalOcean con Docker Compose (API + PostgreSQL), TLS terminado en Cloudflare (SSL Flexible, sin Nginx). Webs en Cloudflare Pages. IA serverless en Cloudflare Workers AI. Despliegue por scp del JAR + docker build remoto.

¿Qué mejorarías con más tiempo?

Proteger /admin y /chat (roles + rate limiting), verificar la firma de Apple, full-text search con GIN en inglés, Redis para blacklist/códigos, SSL Full Strict y webhook de RevenueCat para premium 100 % server-side.
